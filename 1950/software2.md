# SEAC/SWAC 程式設計 — 早期儲存程式方法的實踐

一九五〇年，SEAC 和 SWAC 的程式設計師們在開發美國最早的儲存程式計算機程式的過程中，逐漸形成了一套初步的程式設計方法論。這些方法雖然原始，但它們為後來所有的計算機程式設計實踐奠定了基礎。

SEAC 和 SWAC 的程式設計都是從編碼表開始的。程式設計師需要先確定程式的整體邏輯結構，然後在紙上逐條寫出機器指令。SEAC 使用 45 位元的指令字，SWAC 使用 37 位元的指令字——每條指令都需要手動編碼為二進位或八進位形式。

記憶體管理是程式設計師面臨的最棘手的問題之一。SEAC 只有 512 到 1024 個字的記憶體空間，程式設計師需要精確規劃每個字的用途。程式指令佔用多少空間？資料變數佔用多少？中間結果放在哪裏？這些問題都需要在開始編寫程式之前就規劃清楚。一個常見的解決方案是：將記憶體劃分為固定大小的區塊，每個區塊分配給特定的用途。

SEAC 的程式設計師開發了一種稱為「繪圖板」（plotting board）的除錯方法。他們將程式執行過程中關鍵變數的變化以圖形方式繪製出來，透過視覺化的方式來檢查程式的正確性。這個方法在記憶體有限的條件下非常有效——它讓程式設計師能夠以圖形方式理解程式行為，而不需要逐條指令檢查。

SWAC 的程式設計師則發展出了「檢查點」（checkpoint）的概念。他們在程式的關鍵位置插入特殊的指令，讓程式在執行到這些位置時將暫存器的內容輸出到電傳打字機。這種方法允許程式設計師在程式執行完成後分析其執行軌跡——這是日誌除錯法的前身。

SEAC 和 SWAC 的程式設計環境雖然原始，但已經包含了後來軟體工程的核心元素。編碼表對應於今天的原始碼檔案；記憶體規劃對應於編譯器的記憶體分配；繪圖板對應於視覺化除錯器；檢查點對應於日誌系統。概念的具體形式雖然改變了，但核心挑戰——如何確保程式正確、高效地執行——從未改變。

一九五〇年的 SEAC 和 SWAC 程式設計師還面臨一個後來軟體工程師不必擔心的問題：硬體可靠性。真空管隨時可能故障、記憶體可能丟失資料、紙帶可能撕裂。程式設計師需要設計能夠檢測和恢復硬體錯誤的程式——這個需求催生了最早的錯誤處理和容錯程式設計方法。
