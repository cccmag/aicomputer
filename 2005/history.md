# 歷史穿梭：2005 年的種子

## 處理器架構的兩條岔路

2005 年是處理器設計哲學的分水嶺。兩條路線在同一時間交會：

**路線一：Apple 轉向 Intel**。2005 年宣布的轉換在一年內實現——到 2006 年 MacBook Pro 和 iMac 都已採用 Intel Core Duo。這條路線後來在 2020 年再次轉向 Apple Silicon（M1），但 Intel 時代的 Rosetta 相容性技術為後來的 Rosetta 2 提供了經驗。

**路線二：Arduino 開放硬體**。當 Apple 和 PC 廠商專注於更高效能的 x86 處理器時，Arduino 選擇了另一條路——使用低成本的 Atmel AVR 微控制器，將運算帶入實體世界。這條路線最終導向了物聯網（IoT）革命和自造者運動。

這兩條路線說明了同一個道理：運算的未來不僅僅是更快的處理器，而是運算的普及化和多樣化。

## 版本控制的兩個世界

2005 年是版本控制系統的轉捩點。Git 的誕生代表了一個根本性的變化：

**集中式（CVS/Subversion）** → **分散式（Git/Mercurial）**

Git 的成功不僅在於技術——雖然它的分支合併模型確實優於 Subversion——更在於它所支援的協作模式。GitHub 於 2008 年創立，將 Git 與社交網路元素結合，創造了前所未有的開源協作平台。2005 年 Linus 為了維護 Linux 而寫的這個工具，最終改變了全世界開發者的工作方式。

## Google Maps：從地圖到平台

Google Maps 的 API（2005 年 6 月發布）是早期 Web 2.0 API 的典範。它展示了：
- 開放 API 可以創造生態系統
- 混搭（mashup）文化可以用創新的方式組合既有服務
- 地圖不僅是地圖，而是位置資訊的平台

Google Maps 的 API 模式後來被 Google 應用於其他服務（YouTube API、Google Charts API），也影響了 Facebook 的 Graph API 和 Twitter API。2005 年的地圖服務開啟了「平台化」（platformification）的時代。

## YouTube 的蝴蝶效應

YouTube 在 2005 年的誕生引發了一系列連鎖效應：
1. 影片分享的民主化
2. 公民新聞的崛起
3. 音樂產業的展示管道轉移
4. 長尾內容的經濟學
5. Google 在 2006 年以 16.5 億美元收購——當時 Google 最大的收購案

YouTube 來自一個簡單的洞察：「為什麼分享影片這麼難？」這個問題改變了媒體產業的格局。YouTube 的成功證明了使用者生成內容（UGC）的價值——專業製作的內容不再是人們唯一關心的東西。

## Rails 與 Web 框架的演化

Ruby on Rails 在 2005 年的成功可以追溯為一條清晰的影響鏈：

**Smalltalk（1970s）** → **Ruby（1995）** → **Rails（2004-2005）** → **現代 Web 框架（Laravel、Django、Spring Boot）**

Rails 的「約定優於配置」和「DRY」原則改變了 Web 開發的思考方式。它教會了開發者一個重要的教訓：框架的價值不在於它做了多少事，而在於它讓開發者不必做多少事。

## 2005 年的時代脈絡

- **卡特里娜颶風（2005 年 8 月）**：部落格和公民新聞在災難報導中展現了比主流媒體更快的反應速度和更廣的視角
- **倫敦爆炸案（2005 年 7 月）**：手機攝影和公民記者成為新聞報導的重要組成部分——見證了「我們都是記者」的時代
- **京都議定書生效**：綠色科技和節能成為科技產業的關注焦點，影響了處理器設計的功耗目標
- **Web 2.0 高峰**：從概念到實務，O'Reilly 的 Web 2.0 理念在 2005 年找到了大量實踐者——Digg、del.icio.us、Flickr、YouTube 都是 Web 2.0 的代表
