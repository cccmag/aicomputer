# GitHub 的協作模式：開源社群的進化

## Fork 與 Pull Request

GitHub 在 2007 年的創立不僅僅是程式碼託管，更是一種全新的開源協作模式。傳統的開源專案運作方式是「提交修補檔」（submit patches）——貢獻者下載原始碼、做出修改、產生 diff 檔案、然後透過郵件或論壇提交給維護者。

GitHub 的 fork 與 pull request 機制從根本上簡化了這個流程。任何開發者都可以用一鍵 fork 複製專案，在個人分支上進行修改，然後發送 pull request 請求專案維護者合併變更。維護者可以在 pull request 上逐行審閱代碼、留言討論、要求修改，最後在平台上一鍵合併。

## 網路效應的開端

GitHub 的商業模式建立在網路效應之上。越多的專案託管在 GitHub 上，就越多的開發者會註冊 GitHub；越多的開發者使用 GitHub，它對新專案的吸引力就越大。

2007 年 GitHub 的早期使用者多是 Ruby 社群——因為創辦人 Preston-Werner 和 Wanstrath 都是 Ruby 開發者。但很快地，Python、JavaScript、Java 等社群的專案也開始遷移到 GitHub。

## 開源的民主化

GitHub 最重要的貢獻是開源的民主化。在 GitHub 之前，建立一個開源專案需要自己架設伺服器、設定郵件列表、管理修補檔——這些技術門檻讓許多潛在的創作者卻步。GitHub 將這些基礎設施變成了免費的服務，讓任何人都可以在一分鐘內啟動一個開源專案。
