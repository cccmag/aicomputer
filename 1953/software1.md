# IBM 701 程式設計 — 機器語言與絕對二進位

一九五三年，IBM 701 的程式設計師面對的是一個沒有高階語言、沒有作業系統、甚至沒有組合語言的編程環境。他們直接以機器語言與計算機溝通。

IBM 701 的指令格式為半字（十八位元）或全字（三十六位元）。每條指令包含一個操作碼和一或多個位址。程式設計師需要將運算元和指令直接以二進位或八進位形式寫在編碼表上。然後由打孔卡操作員將這些編碼卡片輸入機器。

程式載入的過程尤為繁瑣。701 沒有內建的載入程式——操作員需要先透過控制台設定一組初始開關，將一個小型的引導程式手動輸入記憶體。這個引導程式再從打孔卡讀入真正的應用程式。如果引導過程出了差錯，整個操作就必須重來。

IBM 701 的軟體工具在初期極其有限。IBM 提供了一套稱為「絕對二進位載入器」（Absolute Binary Loader）的基本工具，能將卡片上的二進位碼載入到指定的記憶體位址。此外還有一個簡易的除錯工具，可以讓操作員在控制台上檢視和修改記憶體內容。

程式設計師在 701 上的工作方式非常獨特。編寫程式時，他們先在編碼表上規劃記憶體佈局，手動分配每個資料和指令的位置。由於沒有符號名稱，所有位址都是絕對數字。跳轉指令的目標位址是一個具體的記憶體位置，而非標籤。這種程式設計方式對人的記憶力和專注力是極大的考驗。

除錯更是考驗耐心。當程式執行失敗時，程式設計師只能依賴控制台上的指示燈——每盞燈對應記憶體中的一個位元——來推斷問題所在。他們可以讓機器以單步模式執行，逐條指令檢查計算結果的變化。這個過程可能需要數小時甚至數天。

雖然 IBM 701 的程式設計環境在今天的標準看來幾乎原始得不可思議，但正是在這種限制中，程式設計師發展出了最早的程式設計方法論：模組化思考、測試驅動的除錯流程、以及結構化的程式組織方式。這些方法論在後來的高階語言環境中被繼承並發揚光大，成為軟體工程的基石。

一九五三年的 IBM 701 程式設計實驗，與其說是技術實踐，不如說是對「人類如何與機器溝通」這一根本問題的探索。這條探索之路最終通向組合語言、FORTRAN、COBOL，以及今日的 Python 和 Rust。
